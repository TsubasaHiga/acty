# スキル・コマンド リファレンス

Claude Code のスキル（`/スキル名`）を使って各ワークフローを実行できます。

---

## 開発

### Issue 対応

```
/work-issues 42
/work-issues
```

Issue 番号を指定して対応を開始します。省略時は Open Issue の一覧から若い番号順にノンストップで進めます。`CLAUDE.md` および `docs/shared-guidelines.md` のガイドラインに従い、ブランチ作成から PR 作成まで一貫して実行します。

### Codex コードレビュー

```
/codex-review
/codex-review セキュリティ重点
/codex-review パフォーマンス重点
```

feature ブランチの変更を Codex プラグイン（`/codex:review`）にレビュー依頼し、指摘がなくなるまで修正を繰り返します。`/work-issues` 内では PR 作成前に自動実行されます。重点ポイント指定時は `/codex:adversarial-review` を使用します。

レビュー結果の分類:

| レベル   | 意味                   | 対応     |
| -------- | ---------------------- | -------- |
| Critical | バグ・セキュリティ問題 | 必ず修正 |
| Warning  | 品質向上の推奨修正     | 修正推奨 |
| Info     | 提案（対応任意）       | 検討     |
| Good     | 良い実装の評価         | 参考     |

その他の Codex コマンド:

| コマンド                    | 用途                                         |
| --------------------------- | -------------------------------------------- |
| `/codex:review`             | 標準レビュー（`--base main` でブランチ比較） |
| `/codex:adversarial-review` | 設計判断・トレードオフを深掘りするレビュー   |
| `/codex:rescue`             | バグ調査やタスク委譲                         |
| `/codex:status`             | 実行中ジョブの進捗確認                       |
| `/codex:result`             | 完了ジョブの結果表示                         |
| `/codex:setup`              | セットアップ・認証確認                       |

### レビューフィードバック対応

```
/handle-review
/handle-review 15
```

PR レビューのフィードバックを確認し、対応の必要性を検討します。各コメントについて「修正必要 / 議論必要 / 対応不要」を判断し、ユーザー確認後に対応を実施します。

### アプリ全体調査

```
/audit-app バグ重点
/audit-app UX改善
/audit-app パフォーマンス
/audit-app デグレ確認
/audit-app
```

アプリ全体を探索し、潜在的なバグ・改善点・UX 課題を調査して Issue に起票します。Claude 調査隊（複数の Explore サブエージェント）と Codex 調査隊（`codex exec`）を並行起動し、両者の結果をクロスチェックして精査した上で起票します。

| ステップ               | 内容                                                           |
| ---------------------- | -------------------------------------------------------------- |
| 調査計画               | 重点ポイントに応じて調査対象モジュールを決定                   |
| 並行調査               | Claude 調査隊と Codex 調査隊を同時に起動                      |
| マージ・クロスチェック | 両者の結果を統合し、片方のみの指摘はオーケストレータが再確認   |
| 精査                   | 妥当性判断（誤検出・過剰指摘の除外）                          |
| Issue 起票             | 既存 Issue との重複排除後、重要度順に起票                      |

## リリース

### 手動テストチェックリスト

```
/manual-test-checklist 0.0.12
/manual-test-checklist
```

指定バージョンタグから最新 main までのコード変更を分析し、手動での動作確認が必要な項目を GitHub Issue として起票します。手動でしか確認できない変更（視覚的変更、レスポンシブ、ページ遷移等）を漏れなくチェックリスト化します。バージョンタグ省略時は直前のタグを自動検出します。

| ステップ           | 内容                                                                         |
| ------------------ | ---------------------------------------------------------------------------- |
| 情報収集           | git diff/log、PR 一覧から変更ファイル・コミットを取得                        |
| 変更分析           | components/layouts/pages/styles/store 等の単位でグルーピング                 |
| テスト除外判定     | 手動テスト必須カテゴリ（UI/レスポンシブ/SEO/インタラクション等）に基づく判定 |
| チェックリスト生成 | 重要度順（Critical → High → Medium → Low）にセクション分けして生成           |
| Issue 起票         | ユーザー確認後に GitHub Issue として起票                                     |

### リリース準備

```
/prepare-release 0.0.13
```

指定バージョンタグから最新 main までの変更内容を確認し、Semantic Versioning に基づいて適切なバージョン番号（major/minor/patch）を提案します。Changelog ドラフトの生成、リリースチェックリストの作成も行います。

### ポストリリース

```
/post-release 0.0.14
```

リリース後の作業を一括実行します。GitHub Release の作成（または更新）と次期バージョンに向けた課題整理を行います。

| ステップ       | 内容                                                                    |
| -------------- | ----------------------------------------------------------------------- |
| GitHub Release | 前回タグからの変更を分析し、リリースノートを作成・更新                  |
| 課題整理       | コードベースを探索し、デグレ確認・潜在バグ・改善点を調査して Issue 起票 |
